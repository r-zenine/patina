# Strategy Artifacts (dev-strategy Output)

Schemas for the 4 core artifacts generated by the dev-strategy skill.

All artifacts are created in `./dev-strategy-[FEATURE-NAME]/`.

---

## 1. `code-context.md`

**Purpose**: Quick reference for relevant existing code. Immutable — created during planning, not modified during execution.

**Schema:**
```markdown
# Code Context for [FEATURE NAME]

## Relevant Classes and Functions

- **ClassName.methodName** (`file_path:line_number`) - Brief description
- **ServiceName.operation** (`src/services/example.js:25`) - What this function does

## Key Files to Reference

- `src/models/` - Data models and validation schemas
- `src/services/` - Business logic and external integrations

## Database Schema References

- **Table/Collection**: `table_name` - Purpose and key fields
- **Relationships**: How data connects to existing entities

## Configuration and Environment

- **Environment Variables**: Required config for this feature
- **Dependencies**: External services or libraries used

## Testing Patterns

- **Unit Test Location**: `tests/unit/[feature]/`
- **Test Utilities**: Shared mocks, fixtures, helpers to use
```

## 2. `context-document.md`

**Purpose**: Immutable reference — behavioral spec, architecture summary, constraints, research findings. Created during planning, not modified during execution.

**Schema:**
```markdown
# Context Document for [FEATURE NAME]

## Behavioral Specification
[Plain description of what to build — 2-3 sentences, no technical jargon]

## Codebase Patterns to Follow
- **Data Access**: [Repository pattern, Active Record, ORM]
- **Business Logic**: [Service layer, Domain models]
- **API Design**: [RESTful, GraphQL]
- **Testing**: [Given/When/Then, Page Object Model]

## Technical Constraints
[Only actual constraints mentioned by user or discovered in codebase]

## Local Repository Skills
[Local skills in .claude/skills/, skills/, .skills/ — descriptions and when to use them]
[If none: "No local skills detected"]

## Research Findings (if research was conducted)
- **Technologies Researched**: [List]
- **Implementation Approach**: [How to integrate]
- **Key Insights**: [Findings that influence implementation]
[See ./research/ directory for full research artifacts]
```

---

## 3. `decision-log.yaml` (strategy/planning variant)

**Purpose**: Mutable record of decisions made during planning. Can be extended during execution.

**Schema:**
```yaml
decisions:
  - id: "001"
    type: planning  # design | planning | architecture
    decision: "[Decision made]"
    rationale: "[Why this choice was made]"
    alternatives_rejected:
      - alternative: "[Option not chosen]"
        reason: "[Why rejected]"
    impact: "[Effect on implementation]"
```

**Note**: This is the planning decision log. Contribution decision logs use a separate YAML file per contribution folder.

---

## 4. `implementation-roadmap.md`

**Purpose**: Phase-by-phase execution plan. References the chosen strategy.

**Standard format** (for TDD and Core-then-Integrate):
```markdown
# Implementation Roadmap for [FEATURE NAME]

## Execution Strategy
**Strategy**: [TDD | Steel Thread | Core-then-Integrate]
**Approach**: [Brief description]

## Phase 1: [Title]
**Description**: [What to implement]

**Objectives**:
- **Implementation**: [Clear execution step]
- **Design**: [Design decision to defer]

**Testing Criteria**:
- [How to verify objective 1]

**Dependencies**: [None, or "Must complete Phase X first"]

**Relevant Local Skills**: [Applicable skills, or "None"]

**Files to Modify**:
- `path/to/file.js` - [What changes]
```

**Steel Thread format**: Use `steel-thread-roadmap-template.md` which adds:
- "Steel Thread Scope" section per phase (end-to-end path, user experience, intentional shortcuts)
- "Refactor for Next Phase" section with Just-In-Time guidelines

## Research Artifacts

Created in `./dev-strategy-[FEATURE-NAME]/research/` when the research step is triggered.

**`research-summary.md`**: High-level findings and recommendations
**`technology-research.md`**: Focused analysis of each technology researched
**`implementation-examples.md`**: Code patterns and practical examples

Research is only generated when ALL three conditions hold:
1. User explicitly names a technology they want to use
2. That technology is not currently in the codebase
3. Integration patterns with existing stack are unclear
